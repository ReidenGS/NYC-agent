# NYC 租房 Agent 业务逻辑（V1.1）

## 1. 项目定位
该项目是一个面向“初到纽约用户”的 AI Agent，目标是帮助用户从“对纽约完全不了解”到“可以明确 1-2 个候选居住区域”。

产品形态采用“咨询问答驱动”，不是强制生成完整报告。
- 用户可随时就某个区域发问（如犯罪、娱乐、交通、租金）
- Agent 按问题即时回答
- 当信息足够时再进入推荐/收敛流程
- 完整报告为可选输出，不是默认必经步骤

## 2. 目标用户（优先级）
目标用户优先级如下：
1. 留学生
2. 短期工作/实习人群（3-12 个月）
3. 新入职年轻人（1-3 年）
4. 家庭用户（带孩子）

## 3. 核心痛点（优先级）
用户核心痛点如下：
1. 信息分散在多个平台，难以统一判断
2. 区域之间缺乏客观可比性
3. 看房执行成本高（路线规划、时间安排、预约组织）

## 4. 核心产品能力（优先级）
MVP 最重要能力排序：
1. 区域对比能力（按用户关心维度动态比）
2. 基于约束的区域初筛
3. 可落地推荐清单（3-5 个区域 + 推荐理由）

补充：
- “区域对比报告”改为可选导出能力
- 默认交互是连续咨询 + 实时更新权重 + 即时回答

## 5. Agent 行动边界
当前边界：分析 + 半自动执行。

Agent 应该做：
- 输出分析结论与推荐
- 回答单点问题（某区域犯罪、娱乐、交通、租金等）
- 提供外部信息入口（地图、房源等）
- 生成中介联系话术与看房执行计划

Agent 不做：
- 直接替用户联系中介
- 直接替用户预约看房

## 6. 数据源范围
### MVP 必须接入
1. NYC 各区域基础信息（安全、交通、生活配套）
2. 地图与通勤数据
3. 租金区间数据（按区域/户型）
4. 真实房源数据
5. 用户讨论/评价信号
6. 公开犯罪数据
7. 娱乐设施/POI 密度数据

### 后续可选
1. 学校与医疗资源数据
2. 历史租金趋势与预测

## 7. 性能、质量与成本目标
- 首次问题响应时延：<= 5 秒
- 多维汇总/推荐响应时延：<= 30 秒
- 推荐有效命中率：>= 80%
- 单次完整咨询成本：< 1 美元
- 项目总预算（Demo/MVP 阶段）：< 10 美元

## 8. 合规与安全要求
以下要求全部必须满足：
1. 明确声明“非法律建议、非合同建议”
2. 不使用受保护敏感属性进行歧视性推荐
3. 安全相关结论必须展示数据来源与时间戳
4. 不确定结论必须说明“不确定/数据不足/仅供参考”，不默认向用户展示数值置信度
5. 保留匿名化日志用于审计
6. 最小化采集用户信息并支持会话删除

## 9. 端到端业务流程（问答驱动）
1. 用户自然语言提问或描述需求（可非常不完整）
2. Agent 先识别是否具备唯一硬性必填：`target_area`（目标地区）
3. 若缺少 `target_area`，Agent 先追问该字段，不进入分析
4. 有 `target_area` 后，按用户当前问题即时回答：
   - 问犯罪：返回该区域附近犯罪率/犯罪数量（含时间范围）
   - 问娱乐：返回附近娱乐设施数量/密度
   - 问交通：返回通勤可达性与时间分布
   - 问租金：返回租金区间与可得房源概况
   - 问天气：返回该区域当前/指定时刻的天气预报
5. 每次回答后，Agent 可追问“该维度在你决策中的权重是多少”
6. 用户回答后，Agent 更新用户权重画像并在后续回答中生效
7. 当关键信息足够时，Agent 提供区域排序与推荐（可选输出完整报告）
8. 用户可继续追问，Agent 持续增量重算，直到收敛到 1-2 个候选区域

## 10. 必填信息与权重管理规则
### 唯一硬性必填（MVP）
- `target_area`（目标地区）

### 建议补充信息（非硬性）
- `budget_monthly`
- `target_destination`
- `max_commute_minutes`
- `lease_term`
- `move_in_date`
- `lifestyle_preferences`

### 权重维度（可由用户动态设置）
- `weight_safety`
- `weight_commute`
- `weight_rent`
- `weight_convenience`
- `weight_entertainment`

规则：
1. 初始使用默认权重（可配置）。
2. 用户提到“更在意某项”或明确给出比例时，覆盖对应权重。
3. 若用户只给相对偏好（如“安全最重要”），系统做归一化映射并回显。
4. 每次权重变更都需要可视化展示并允许用户一键纠正。

## 11. A2A 多 Agent 角色设计
### Orchestrator Agent（编排 Agent）
- 管理会话状态（已知字段、缺失字段、权重、历史问答）
- 决定“直接答复”还是“追问补齐”
- 合并子 Agent 结果并输出最终答案

### Neighborhood Agent（区域画像 Agent）
- 构建区域画像（安全、便利、娱乐）
- 对安全/便利/娱乐类问题做领域内查询规划

### Transit Agent（通勤 Agent）
- 计算通勤可达性与稳定性

### Weather Agent（天气 Agent）
- 处理目标区域当前天气、小时级天气预报和指定时刻天气问题
- 不参与住房推荐打分，只作为生活辅助信息展示

### Housing Agent（租房 Agent）
- 分析租金区间与房源可得性

### Profile Agent（会话画像 Agent）
- 管理 session、槽位、权重、偏好和短对话摘要

### Decision Agent（决策 Agent）
- 使用当前权重做多维打分
- 生成可解释推荐理由

## 12. MCP 工具层设计
规划 MCP 工具如下：
1. `mcp-housing`：租房数据只读 SQL 校验执行、租金/房源数据访问
2. `mcp-safety`：安全/犯罪数据只读 SQL 校验执行、安全地图图层数据访问
3. `mcp-amenity`：便利设施数据只读 SQL 校验执行、便利设施点位访问
4. `mcp-entertainment`：娱乐设施数据只读 SQL 校验执行、娱乐点位访问
5. `mcp-transit`：MTA 实时通勤、GTFS-RT、Bus Time、站点查询
6. `mcp-weather`：National Weather Service 小时级天气预报、天气缓存
7. `mcp-profile`：用户偏好、权重、会话状态和推荐结果读写

每个 MCP 返回都必须包含：
- `source`
- `timestamp`
- `confidence`

补充规则：
- `housing-agent` 和 `neighborhood-agent` 可以基于已注入的领域表结构生成只读 SQL。
- MCP 不负责生成 SQL，只负责校验 SQL 安全性并用只读账号执行。
- 若当前表结构无法支持用户想查的数据，Domain Agent 不调用 MCP，直接返回无法支持原因。
- SQL 查询成功但无匹配结果时，Agent 只能说明“当前数据库没有找到匹配数据”，不能说现实中一定不存在。

## 13. 自然语言理解与自动追问机制
系统必须支持“纯自然语言”接入。用户可以使用口语化、跳跃式、不完整表达。

抽取槽位：
- 必填槽位：`target_area`
- 非必填槽位：预算、通勤、租期、入住时间、生活偏好等
- 权重槽位：安全/通勤/租金/便利/娱乐

追问优先级：
1. 第一优先永远是 `target_area`（若缺失）
2. 其次是当前问题直接相关的缺失信息（例如问通勤但未给目的地）
3. 再其次是权重澄清（提升后续推荐质量）

追问策略：
1. 一次只追问一个问题
2. 优先问“对当前问题影响最大”的字段
3. 最多 3 轮连续追问，避免打断用户
4. 每轮回答后执行增量抽取，不重置既有信息

冲突处理：
- 检测到冲突值（例如两个不同预算）时先澄清再更新
- 重要更新必须回显（尤其是权重）

## 14. 前端交互与可视化状态要求
前端必须提供“信息与权重状态面板”，用于让用户一目了然确认 Agent 理解是否正确。

面板包含两块：
1. 必填信息状态
   - `target_area`：已填写/未填写
   - 未填写时固定提示“请先提供目标地区”
2. 权重状态
   - 显示当前各维度权重（安全/通勤/租金/便利/娱乐）
   - 显示来源（默认值/用户明确指定/Agent 推断）
   - 支持用户直接修改并即时生效

交互要求：
- 每次 Agent 更新权重后，前端高亮变化项
- 用户可一键“确认权重”或“恢复默认”
- Agent 回答时引用当前权重快照，避免“答非所问”

## 15. 用户侧输出规范
1. 默认输出：针对当前问题的即时答案（不强制长报告）
2. 可选输出：区域对比摘要/完整报告
3. 推荐输出：当信息充分时给出 3-5 区域清单并解释
4. 执行输出：房源清单、看房顺序、中介联系话术
5. 合规页脚：免责声明、数据来源、数据时间戳；低置信度时用自然语言说明不确定原因

## 15A. 天气功能业务逻辑
天气功能是生活辅助能力，不作为区域推荐的核心权重，不影响安全、通勤、租金、便利、娱乐五个决策权重。

触发方式：
1. 默认展示：当会话中已有 `target_area`，前端可以展示该区域天气卡片。
2. 用户主动询问：例如“Astoria 今天晚上会下雨吗”“明早出门冷不冷”。
3. 指定时刻询问：Agent 从自然语言中抽取 `target_time`，查询该时刻附近的小时级预报。

槽位规则：
- 必填：`target_area`
- 可选：`target_time`
- 如果用户没有给地区，但会话已有 `target_area`，使用会话地区。
- 如果用户没有给地区且会话也没有 `target_area`，先追问“你想查哪个区域的天气？”
- 如果用户没有给时间，默认返回当前到未来 6 小时概况。

回答内容：
- 温度
- 天气摘要
- 降水概率
- 风速/风向
- 预报时间范围
- 数据来源和更新时间

数据策略：
- 使用 National Weather Service API。
- NWS 不需要 API Key，但请求必须带 `User-Agent`。
- `/points/{lat},{lon}` 的网格映射可缓存 7 天。
- 小时级 forecast 可缓存 30-60 分钟。
- 天气 API 失败不阻塞住房、区域、通勤等核心回答。

## 16. 1 周 MVP 交付计划（单人 + AI Coding）
1. Day 1：定义会话状态模型（必填字段、槽位、权重）、FastAPI 骨架
2. Day 2：实现问题分类与按需查询（犯罪/娱乐/交通/租金）
3. Day 3：接入 Safety/Map/Housing 数据并统一返回格式
4. Day 4：实现自动追问与权重更新闭环
5. Day 5：实现前端“必填项+权重状态面板”
6. Day 6：做性能/成本优化（缓存、超时、降级）
7. Day 7：准备 Demo（重点展示问答驱动+权重可视化）

### 16.1 租房数据执行规则
为控制 Demo 成本并保证响应稳定，租房数据不作为高频实时 API 查询能力。

执行规则：
1. RentCast 作为离线/低频数据采集源，用于手动导入或定时刷新目标区域的租金与房源快照。
2. 用户咨询租金、房源数量、典型价格区间时，默认读取 PostgreSQL 中已缓存的租房数据。
3. Agent 不应在每次用户提问时直接调用 RentCast，避免快速耗尽免费额度或产生不可控成本。
4. 若数据库中的租房数据过旧，Agent 需要在回答中标注数据时间戳，并提示“当前租房数据可能不是实时数据”。
5. 当用户明确要求刷新最新租房数据时，Agent 可以说明该操作会消耗外部 API 调用额度，再执行受控刷新。
6. 租金基准优先使用免费/公开数据源兜底，例如 HUD FMR、Zillow ZORI；RentCast 主要用于真实房源与市场快照增强。

## 20. 加分功能（可选）：地图可视化决策层
该功能为加分项，不作为 MVP 上线阻塞条件。若时间不足可暂不实现。

### 20.1 目标
在文本数据之外，用地图可视化展示目标区域及周边指标分布，帮助用户更直观理解“哪里更安全、哪里娱乐设施更多、哪里通勤更方便”。

### 20.2 核心展示能力
1. 区域着色（Choropleth/Heatmap）：
   - 犯罪率高：区域颜色向红色过渡
   - 犯罪率低：区域颜色向绿色或浅色过渡
2. 设施点位可视化（Marker Layer）：
   - 娱乐设施用蓝色点标记
   - 点位数量越多，视觉密度越高（可结合聚合）
3. 多指标图层切换：
   - 安全图层
   - 娱乐图层
   - 交通图层
   - 租金图层
4. 指标悬浮信息：
   - 鼠标悬停显示该区域关键数值（如犯罪数、娱乐设施数量、平均租金、平均通勤）

### 20.3 映射规则（可配置）
1. 颜色映射：
   - 将指标做标准化（min-max 或分位数）后映射颜色
   - 避免绝对值跨度过大导致颜色失真
2. 点位映射：
   - 娱乐设施按 POI 分类过滤后上图
   - 点过密时采用聚类（cluster）展示
3. 图层互斥与叠加：
   - 默认单图层，避免信息过载
   - 允许高级用户开启双图层叠加

### 20.4 与 Agent 业务联动
1. 用户问“这个区犯罪怎么样”时：
   - 文本回答给数值
   - 地图自动切到安全图层并高亮目标区域
2. 用户问“娱乐设施多不多”时：
   - 文本回答给数量/密度
   - 地图显示蓝色娱乐点位分布
3. 用户修改权重后：
   - 地图可切换到“综合推荐视图”
   - 高分区域颜色更突出

### 20.5 前端实现建议（可选技术）
- 地图库：Mapbox GL JS 或 Leaflet
- 热力层：Heatmap/Fill Layer
- 点位层：Circle/Marker + Cluster
- 交互：图层开关、图例、悬浮信息卡片

### 20.6 降级策略（来不及时）
若一周内无法完成完整地图模块，按以下优先级降级：
1. 仅实现“目标区域高亮 + 指标 tooltip”
2. 再补“安全热力着色”
3. 最后补“娱乐点位密度和多图层切换”

该降级不影响主业务闭环（问答、追问、权重、推荐）。

## 21. 实时通勤功能（MVP）
该功能用于帮助用户规划实际出发时间，而不是替代长期选区评分。

### 21.1 核心目标
当用户已经有目标居住区域、具体出发地址或指定站点时，Agent 可以回答：
- 下一班/下两班地铁或公交什么时候到/出发
- 从出发点走到站点需要多久
- 推荐什么时候出门
- 预计什么时候到达目的地
- 全程大约需要多久

### 21.2 输入规则
支持以下输入：
1. “从某地到某地，现在怎么走？”
2. “某站某线路下一班什么时候来？”
3. “从我的目标区域到目的地，下一班地铁/公交什么时候？”
4. “从某地到某地，坐地铁/公交要多久能到？”

出发点规则：
- 用户给具体地址或站点：优先使用用户指定地点
- 用户只给 `target_area`：Agent 需要追问出发地点
- 站点候选不唯一：Agent 追问用户确认

目的地规则：
- 本轮用户明确说目的地：优先使用本轮目的地
- 本轮没说：使用会话中的 `target_destination`
- 会话中也没有：Agent 追问目的地

交通方式规则：
- 用户明确说地铁/公交：只查对应方式
- 用户没说交通方式：追问“你想查地铁还是公交？”
- 用户说“都可以”：地铁和公交都查，返回更合理的一种并说明依据

### 21.3 MVP 范围
MVP 支持：
- 地铁
- 公交
- 步行接驳
- 单线路实时下一班/下两班

加分增强：
- 多线路换乘
- 公交与地铁组合路线
- 地图路线展示

### 21.4 返回内容
实时通勤回答必须包含：
1. 下一班/下两班车时间
2. 推荐出发时间
3. 步行到站时间
4. 预计到达目的地时间
5. 总通勤时间
6. 数据是否实时

前端展示：
- 文字回答
- 下一班/下两班小卡片
- 简单路线时间轴：步行 -> 等车 -> 乘车 -> 到达
- 地图路线展示作为后续增强

### 21.5 数据策略
使用“按需请求 + 短缓存”：
- 用户问实时通勤时才请求 MTA API
- 对同一个 `stop_id + route_id + direction` 缓存 `30-60秒`
- 缓存命中直接返回
- 缓存过期后重新请求 MTA
- 保存匿名查询日志，用于 Demo 指标和工具调用分析

实时通勤不进入长期区域推荐打分。  
区域推荐打分使用静态/常规通勤时间；实时下一班车只用于用户主动查询和出发规划。

### 21.6 降级策略
如果实时数据不可用：
1. 使用静态 GTFS 或常规通勤时间
2. 明确标注“非实时”
3. 给出最近站点和建议出发窗口
4. 不承诺下一班车的实时准确性
